# SPEC.md - ゲームエンジン実装仕様

この文書は、本プロジェクトのゲームエンジン実装仕様を定義する。  
採用ルールは `RULE.md` に定義し、この文書はそれをどのように実装するかを定義する。

**実装と最終的に一致していなければならない正本はこの `SPEC.md` とする。**  
`RULE.md` と矛盾がある場合は、両者を更新して整合させること。

---

## 1. 目的

日本式リーチ麻雀の対局シミュレータを C++20 で実装する。  
主目的は以下の 3 点である。

1. **高速自己対戦**に耐えること
2. **厳密で再現可能な対局進行**を提供すること
3. **探索・強化学習・将来の Python 連携**に適した API と状態管理を提供すること

初期実装は **L2** を対象とし、L2 の成立を最優先する。  
L3 / L4 は将来拡張とする。

## 4. 実装方針

## 4.1 基本原則
- ルール定義とエンジン仕様を分離する
- コア状態は **値コピー可能** にする
- 状態遷移は **deterministic** にする
- 同一 seed + 同一 action 列から常に同一結果を得られるようにする
- 例外に依存せず、エラーコードで扱う
- Debug / Fast の 2 モードを持てる設計にする
- ログ生成や重い検証は無効化可能にする
- 役判定・流局判定・精算は後から拡張しやすい構造にする

## 4.2 状態複製性
- `MatchState`、`RoundState`、`EnvironmentState` は **値コピーで完全複製可能** であること
- 複製後の状態は元状態と独立に進行できること
- RNG 内部状態も複製対象に含めること
- グローバル可変状態を持たないこと

## 4.3 不完全情報の扱い
- エンジン内部状態は **完全情報状態** として保持する
- エージェント向け観測は **部分観測** として生成する
- determinization は原則として探索側で行う
- エンジンは determinization を支援する補助 API を提供してよい

## 6. データモデル

## 6.1 牌表現

### 6.1.1 内部表現
牌は以下の 2 系統を併用する。

1. **136 枚の実牌 ID**
2. **34 種インデックス**

用途は以下のとおり。

- 山、配牌、手牌、実際の打牌・副露の管理: 136 枚実牌 ID
- 和了判定、役判定、符計算、受け入れ判定: 34 種インデックス集計

### 6.1.2 赤牌
- 赤5萬、赤5筒、赤5索は通常牌と別の実牌 ID を持つ
- 34 種集計上は通常の 5 と同一種別として扱う
- 赤ドラ判定は実牌 ID から判断する

### 6.1.3 表示順
ログ・CLI・デバッグ表示では以下の順にソートする。

- 萬子 → 筒子 → 索子 → 字牌
- 同数では通常牌を先、赤牌を後

## 6.3 Action

Action は **完全な構造体** として表現する。  
整数 ID のみで表現してはならない。

必須情報:

- `ActionType`
- actor（行動者プレイヤー ID）
- tile（対象牌。必要な場合）
- target_player（必要な場合）
- meld_type（必要な場合）
- `riichi` フラグ（打牌と同時宣言用）
- 補助パラメータ（必要に応じて）

代表的な `ActionType`:

- `Discard`
- `TsumoWin`
- `Ron`
- `Chi`
- `Pon`
- `Daiminkan`
- `Kakan`
- `Ankan`
- `Skip`

### 6.3.1 打牌と立直
- 立直宣言は **`Discard` + `riichi=true`** で表現する
- `RiichiDiscard` のような独立 `ActionType` は持たない
- ツモ切りは独立アクションにせず、通常打牌の一種として扱う

状態遷移の結果として発生した事象は Event として記録する。  
`step()` は必要に応じて Event 列を返す。

代表的な Event:

- `RoundStartEvent`
- `DealEvent`
- `DrawEvent`
- `DiscardEvent`
- `RiichiEvent`
- `ChiEvent`
- `PonEvent`
- `KanEvent`
- `DoraRevealEvent`
- `RonEvent`
- `TsumoEvent`
- `AbortiveDrawEvent`
- `ExhaustiveDrawEvent`
- `RoundEndEvent`
- `MatchEndEvent`

## 6.5 ErrorCode

エラーは原則として `ErrorCode` で返す。  
例外に依存しない。

代表例:

- `Ok`
- `IllegalAction`
- `WrongPhase`
- `InvalidTile`
- `InvalidActor`
- `InconsistentState`
- `ActionNotAllowedInFastMode`（必要なら）
- `UnknownError`

### 6.5.1 違法手
- `step()` に違法手が渡された場合、**デバッグでは assert、通常はエラーコードを返す**
- 失敗時は **状態不変** を原則とする

## 7. フェーズモデル

エンジン内部は明示的な phase machine として実装することを推奨する。

最小限必要な phase:

- `StartMatch`
- `StartRound`
- `DrawPhase`
- `SelfActionPhase`
- `ResponsePhase`
- `ResolveResponsePhase`
- `ResolveWinPhase`
- `ResolveDrawPhase`
- `EndRound`
- `EndMatch`

## 7.1 SelfActionPhase
現在手番プレイヤーが自分の行動を選択するフェーズ。

例:
- 打牌
- 立直打牌
- 暗槓
- 加槓
- ツモ和了

## 7.2 ResponsePhase
他家打牌に対して応答可能なプレイヤーが行動を選択するフェーズ。

応答候補:
- ロン
- ポン
- 大明槓
- チー
- スキップ

## 7.3 Response 解決
他家打牌後は **全員分の応答候補を列挙** し、以下の優先順位で解決する。

1. ロン
2. ポン / 大明槓
3. チー

- ロンは複数人同時成立可
- ポン / 大明槓競合時は、**放銃牌を捨てたプレイヤーから見て先順位のプレイヤーを優先** して一意に解決する
- 実装では、この優先順を常に同じ規則で決定しなければならない
- チーは最優先権を持つ場合のみ成立する

## 8. API 仕様

## 8.1 環境 API
中心 API は以下とする。

- `reset(...)`
- `step(...)`
- `get_legal_actions(...)`
- `make_observation(...)`

## 8.2 reset()

### 8.2.1 半荘モード
- 半荘全体を初期化する
- 未指定時は seed に基づき起家を決定する
- 起家は外部指定も可能にする

### 8.2.2 1局モード
以下を外部から注入可能とする。

- 配牌
- 局番号
- 親
- 本場
- 供託
- ドラ表示
- 山順
- 点数
- 必要に応じたその他初期状態

テスト・デバッグ用途で局面再構築しやすいことを重視する。

## 8.3 step()

`step()` は少なくとも以下を返せる構造にする。

- 次状態（または内部状態更新結果）
- 報酬
- 終了フラグ
- エラーコード
- 実行されたイベント列

### 8.3.1 粒度
- 基本は **現在行動権を持つ主体の 1 行動** を進める粒度とする
- ResponsePhase のみ、応答可能者の action 群を受け取れる設計を許可する

## 8.4 get_legal_actions()
- 現在フェーズと状態に応じた合法手を返す
- 返り値は `std::vector<Action>` とする
- 各 Action は完全構造体であること

## 8.5 make_observation()
- 部分観測版と完全観測版を提供する
- 返り値型は `PartialObservation` / `FullObservation` を分ける

## 9. Match / Round / Turn の分離

## 9.1 MatchState
半荘全体の管理責務を持つ。

- 局番号進行
- 親移動
- 点棒管理
- 本場・供託管理
- 延長局進行
- 半荘終了判定

## 9.2 RoundState
1局内部の進行責務を持つ。

- 山
- 手牌
- 河
- 副露
- ドラ
- 立直状態
- 現在手番
- 直前打牌
- 応答待ち文脈
- 局終了理由

## 9.3 TurnContext / ResponseContext
現在何を解決中かを明示する。

例:
- 通常手番か
- 他家打牌後か
- 応答可能者一覧
- ロン候補者一覧
- 解決対象牌

## 10. 合法手列挙仕様

## 10.1 基本原則
- すべての合法手は明示的に列挙可能であること
- phase ごとに合法手列挙ロジックを分けること
- RL・探索用途のため、候補漏れがないことを重視する

## 10.2 SelfActionPhase での合法手
状況に応じて以下を含む。

- 打牌
- 立直打牌
- 暗槓
- 加槓
- ツモ和了

## 10.3 ResponsePhase での合法手
他家打牌に対して以下を列挙する。

- ロン
- ポン
- 大明槓
- チー
- スキップ

## 10.4 和了は自動処理しない
- 和了可能な場合でも、合法手列挙に `Ron` / `TsumoWin` を含める
- 和了判断は行動として明示する

## 11. ルール処理の責務分離

## 11.1 AgariDetector
- 和了形の判定
- 必要な面子構成列挙
- L2 で必要な通常面子手を対象にする

## 11.2 YakuCatalog / Yaku 判定
- 採用役一覧を管理する
- 各役判定は独立モジュール化する
- 将来追加しやすい構造を維持する

## 11.3 FuCalculator
- 符計算を厳密に行う
- 門前ロン加符、ツモ符、雀頭符、面子符などを扱う

## 11.4 ScoreCalculator
- 翻数 + 符数から点数を計算する
- 高点法で最良構成を選ぶ
- 切り上げ満貫なし
- 積み棒・供託処理の基礎値を提供する

## 11.5 AbortiveDrawRule / EndConditionRule
- 九種九牌判定
- 通常流局
- 親連荘
- 延長局
- 飛び終了
などを扱う

## 12. L2 の和了判定方針

## 12.1 L2 の役セット
L2 では `RULE.md` に記載された L2 対象役のみ判定する。

## 12.2 未実装役
- 未実装役だけで和了できる手は、L2 では **和了不可**
- これは **L2 の暫定仕様** であり、L3 / L4 では役追加に応じて和了可能範囲を拡張する
- つまり L2 は「部分ルール実装」であり、完全実装ではない

## 12.3 高点法
- 複数の面子構成がある場合、実装済み役・符計算の範囲で最も高い点数を採用する

---

## 13. 精算処理

## 13.1 単独和了
`RULE.md` に従って通常のロン / ツモ精算を行う。

## 13.2 複数和了
- ダブロン / トリプルロンを許可する
- 供託棒は **最優先和了者 1 人が総取り**
- 積み棒は **各和了者にそれぞれ加算**
- 親が和了者に含まれる場合は連荘

## 13.3 通常流局
- ノーテン罰符 3000 点を処理する
- 形式テンパイを有効とする
- 親テンパイで連荘、ノーテンで親流れ

## 13.4 九種九牌
- 配牌後の第一ツモ時点で宣言可能
- 親は流れる
- 供託・積み棒は持ち越す

---

## 14. 終局判定

## 14.1 基本
南4局終了時に終局条件を判定する。

## 14.2 和了止め / 聴牌止め
- オーラス親がトップであり、規定条件を満たす場合に終了できる

## 14.3 延長局
- 南4局終了時、トップが 30,000 点未満なら延長局へ進む
- 延長は 1 回のみ

## 14.4 延長局終了
- 延長局では誰かが 30,000 点以上に到達した時点で終了

## 14.5 飛び
- いずれかが 0 点未満になったら終了
- 0 点ちょうどは継続

---

## 15. Observation 仕様

## 15.1 PartialObservation
エージェントに通常渡す観測。少なくとも以下を含む。

- 自家手牌
- 自家副露
- 全員の河
- 全員の公開副露
- ドラ表示
- 局情報
- 本場
- 供託
- 点数
- 現在手番
- phase
- 自家の立直 / フリテン等の状態

## 15.2 FullObservation
デバッグ・学習補助・探索補助用。以下を含む。

- 全員の手牌
- 山順
- 王牌
- 裏ドラ候補
- 内部状態に準じる全情報
- デバッグ・学習補助・探索補助用として、対局進行に必要な全公開・非公開情報を含めてよい
- ただし、RNG の内部状態のような実行制御情報は Observation には含めない

## 15.3 観測と内部状態の分離
- 内部状態そのものを通常エージェントへ渡してはならない
- PartialObservation は常にプレイヤー視点情報のみを含む

## 16. Reward 仕様

初期実装では reward モードを複数サポートする。

## 16.1 PointDelta モード
- 局中は 0
- 和了 / 放銃 / 流局精算 / 半荘終了時に点数差分を返す

## 16.2 FinalRank モード
- 半荘終了時にのみ最終順位に応じた報酬を返す

## 16.3 Combined モード
- 上記の組合せを許可する

実装では RewardPolicy の差し替えで扱える構造を推奨する。

## 17. ログ / リプレイ仕様

## 17.1 ログの原則
- 初期実装で 1 手ごとのイベントログを提供する
- ただし高速探索時は無効化可能にする

## 17.2 最低限記録する情報
- seed
- 局情報（局番号、本場、供託）
- 配牌
- 各ツモ
- 各打牌
- 鳴き
- 立直
- 槓
- ドラ公開
- 和了
- 流局
- 精算結果
- 半荘終了結果

## 17.3 ログ形式
- 内部では C++ の struct ベースで保持する
- 必要なら後で JSON 等へ変換可能にする

## 17.4 リプレイ再現
- seed + action 列から同一結果を再現可能であること
- 可能なら状態スナップショットからの再開もサポートする

---

## 18. MCTS / 探索補助仕様

## 18.1 状態複製
- `EnvironmentState` / `MatchState` / `RoundState` は値コピーで完全複製可能であること
- 複製には RNG 状態を含めること

## 18.2 determinization
- determinization は原則として探索側で行う
- エンジン内部は完全情報状態を持つ
- 観測は部分観測として生成する

## 18.3 補助 API
以下の補助機能を持てる設計にする。

1. 現在局面の完全状態コピー取得
2. 観測プレイヤー視点での未公開牌一覧取得
3. 他家手牌・山順を差し替えた determinized state 構築補助
4. 構築後の状態整合性チェック

## 18.4 グローバル状態
- 探索のため、グローバル可変状態は禁止
- static な可変作業領域に依存しないこと

## 18.5 高速探索
- Fast モードではログ・重い検証を無効化できること
- ルール結果は Debug モードと一致すること

---

## 19. RNG 仕様

- 乱数生成器は `Environment` が保持する
- `reset(seed)` ごとに再初期化する
- 起家決定、山順、配牌、ドラ表示、以降の進行が seed により再現可能であること
- 同じ初期状態コピーから同じ action 列を与えれば同じ結果になること

---

## 20. 状態保存 / 復元

初期実装で、GameState を丸ごと保存・復元可能とする。

用途:
- デバッグ
- テスト
- リプレイ
- 探索分岐
- 学習再開

## 21. Debug / Fast モード

## 21.1 Debug モード
- 整合性チェックを積極的に行う
- 詳細ログを許可する
- assert を活用する

## 21.2 Fast モード
- 高速自己対戦・探索向け
- ログを無効化可能
- 重い整合性検証を抑制可能
- ただしルール結果は Debug モードと一致しなければならない

---

## 22. 状態整合性検証

`StateValidator` 相当の機能で少なくとも以下を検証可能にすること。

- 136 牌の総数整合
- 牌重複なし
- 山枚数整合
- 各プレイヤー手牌枚数整合
- 副露構成整合
- ドラ表示枚数整合
- phase と現在手番の整合
- 九種九牌・和了・流局フラグの矛盾なし

Debug モードでは重要局面で検証を行ってよい。

## 26. 実装上の推奨事項

- 動的メモリ確保を減らす
- 固定長配列や軽量コンテナを優先する
- ログや観測生成を毎回強制しない
- `step()` 中の不要なコピーを避ける
- ただし状態複製のしやすさは損なわない
- 可読性より高速性を優先する箇所は、コメントで理由を残す
