# RL_RULE.md - 学習方針定義

この文書は、このプロジェクトにおける **強化学習・探索・実験方針** を定義する。  
ここに記載された内容は、学習システムが何を目的とし、どのような段階で何を学習対象とするかを定める。

この文書は **学習方針・研究方針** を扱う。  
学習システムの実装仕様、データ形式、worker/learner 構成、保存形式、Python/C++ 境界などの実装詳細は `RL_SPEC.md` を参照する。

ゲームルールそのものは `GAME_RULE.md`、ゲームエンジンの実装仕様は `GAME_SPEC.md` を参照する。

---

## 1. 目的

このプロジェクトの最終目的は、**完全な日本式リーチ麻雀ルールで動作する強い麻雀エージェント** を構築することである。  
ただし、その達成は一足飛びではなく、以下の方針で段階的に進める。

- 学習問題を段階的に簡略化して立ち上げる
- ゲームエンジンと学習システムを分離して設計する
- 実験可能性と比較可能性を重視する
- 単一の最終モデルだけでなく、**試行錯誤の過程自体を個人研究として記録・比較可能** な基盤を作る
- 将来的に Python 学習 + C++ 高速推論へつなげられる構造を前提とする

---

## 2. 基本方針

- 最終目標は **完全行動・完全ルール** の麻雀エージェントである
- ただし初期段階では、学習対象行動とルール適用を制限して学習を成立させる
- 初期段階では、ゲームエンジンの完全性よりも **学習・探索・比較基盤を早く成立させること** を重視する
- その後、学習基盤を保ったままルールと行動を段階的に拡張する
- 学習方針は固定しすぎず、複数モデル・複数特徴量・複数報酬設計を比較できるようにする

---

## 3. 学習段階の定義

学習段階は、番号と名前を併記して管理する。

### Stage 1 (DiscardOnly)
- 自摸直後の打牌のみを学習対象とする
- 副露は発生させない
- ロン / ツモ和了は自動実行とする
- テンパイ時は自動立直とする
- 九種九牌は発生させない
- 通常流局は通常どおり発生する
- 最初の強化学習立ち上げと基盤検証のための段階

### Stage 2 (RestrictedAction)
- 打牌以外の一部行動を段階的に追加する
- 立直判断、和了判断、見逃し判断などを候補とする
- Full → Partial の蒸留実験をこの段階で行ってよい
- 探索の実用性評価を進める

### Stage 3 (FullAction)
- 打牌・副露・立直・和了・スキップを含む完全行動を対象とする
- ゲームエンジン側の完全ルール化と並行して拡張する
- 最終的な実戦的麻雀エージェントの段階

---

## 4. Stage 1 の正式仕様

Stage 1 は、最初に成立させるべき学習問題として以下を正式仕様とする。

### 4.1 学習対象行動
- 自摸直後の打牌のみ

### 4.2 自動処理される行動
- ロン可能なら常にロンする
- ツモ和了可能なら常にツモ和了する
- テンパイ時は常に立直する

### 4.3 無効化する行動・イベント
- 副露しない
- 九種九牌は発生させない

### 4.4 通常どおり残すもの
- 通常流局
- 点数計算
- 本場・供託
- 半荘終了
- 延長局
- 飛び終了

### 4.5 目的
- 打牌 policy 学習を単純化する
- 学習ループ、報酬、観測、記録、評価の基盤を安定させる
- 将来の複雑行動追加に備えた比較基準を作る

---

## 5. ポリシー分割方針

麻雀の行動は性質が大きく異なるため、**単一ポリシーですべての行動を扱うことを前提としない**。

将来的には、少なくとも以下のような責務分割を想定する。

- `DiscardPolicy`
- `CallPolicy`
- `RiichiPolicy`
- `WinPolicy`
- 必要に応じて `AbortiveDrawPolicy`

### 初期段階
- Stage 1 では `DiscardPolicy` のみを学習対象とする
- 他の行動はルール固定またはサブポリシー固定とする

---

## 6. Observation 方針

### 6.1 最終目標
最終的な実戦エージェントは **PartialObservation** を前提とする。  
実戦で使えない情報に依存した推論を最終目標とはしない。

### 6.2 Stage 1 の扱い
Stage 1 では **FullObservation / PartialObservation の両方を正式に扱える設計** とする。  
ただし、**推奨開始点は FullObservation** とする。

理由:
- 学習ループの立ち上げを容易にする
- 価値関数学習を安定させやすい
- エンジン・特徴量・モデル・報酬基盤の不具合切り分けがしやすい

### 6.3 FullObservation の位置づけ
FullObservation は以下の目的で利用してよい。

- 学習基盤の検証
- 上限性能の比較
- teacher モデルの学習
- Full → Partial 蒸留実験

### 6.4 PartialObservation の位置づけ
PartialObservation は以下の目的で利用する。

- 最終的な実戦用 policy 学習
- 不完全情報下の探索
- 実戦性能評価

---

## 7. 特徴量・モデル比較の方針

このプロジェクトは、単一アーキテクチャに固定された学習器ではなく、**比較可能な研究基盤** を目指す。

そのため、以下を前提とする。

- Observation と Feature 表現を分離する
- FeatureEncoder を複数保持できるようにする
- モデルアーキテクチャを複数保持できるようにする
- 実験ごとに encoder / model / reward / self-play 方針を切り替えられるようにする

### 初期に比較対象として想定するもの
- フラット特徴量 + MLP
- チャネルテンソル + CNN

将来的には token 系や hybrid 系を追加してよい。

---

## 8. 学習アルゴリズム方針

### 8.1 固定しすぎない方針
学習アルゴリズムは固定しすぎず、比較可能にしておく。  
ただし、初期の推奨方針は明記する。

### 8.2 推奨
- 初期の推奨は **policy + value** を持つ Actor-Critic 系
- 特に **PPO 系を第一候補** とする

### 8.3 許容
- 初期の軽い模倣学習
- Actor-Critic 系の他バリアント
- 実験目的の簡易 supervised / imitation 学習

### 8.4 Stage 1 の推奨開始方法
- **ルールベース打牌の軽い模倣を少し行った後に self-play へ入る** 方法を推奨する
- ただし、最初から self-play のみで始める実験も許可する

---

## 9. 報酬方針

### 9.1 基本方針
麻雀は報酬が疎で長期依存が強いため、最初から **中間報酬を含む構成** を許可する。

### 9.2 初期推奨
初期の推奨報酬は以下の併用とする。

- **局終了時の点数差分**
- **半荘終了時の最終順位または最終収支**

### 9.3 注意点
- 中間報酬を過剰に入れすぎると局所最適化の危険がある
- 最終順位だけだと学習信号が遠すぎる可能性がある
- そのため、局収支と半荘結果を併用する設計を推奨する

### 9.4 Value 学習
value は複数種類を持てる設計を前提とする。  
初期の推奨は以下の複数ヘッド拡張可能設計とする。

- 局点差 value
- 半荘最終収支 value
- 将来的に順位 value も追加可能にする

---

## 10. self-play 方針

### 10.1 基本方針
self-play は、学習中ポリシーだけでなく **ベースラインエージェントを混ぜた対戦** を許可する。

### 10.2 初期相手構成
初期は以下の 2 系統を主に使う。

- 学習中ポリシー
- ルールベースベースライン

### 10.3 推奨初期値
相手構成比率は **設定可能** としつつ、推奨初期値を定める。

推奨初期値:
- 学習中ポリシー: 0.5
- ルールベースベースライン: 0.5

### 10.4 将来の変更
学習が進んだら、学習中ポリシーの比率を高めてよい。  
また、評価用には過去 checkpoint や別ベースラインを混ぜてよい。

---

## 11. ルールベースベースライン方針

### 11.1 目的
- 学習初期に和了・テンパイの発生率を高める
- 学習信号を得やすくする
- 比較対象を作る

### 11.2 初期実装の最小要件
- **シャンテン数最小** を目指す打牌

### 11.3 推奨目標
将来的な推奨ベースラインは以下とする。

- シャンテン数最小
- 同点なら受け入れ最大
- テンパイ時は自動立直

### 11.4 実装段階の扱い
初期実装では最小要件のみでもよい。  
その後、受け入れ最大や立直自動化を追加してよい。

---

## 12. 探索方針

### 12.1 段階的導入
探索は次の順で導入することを推奨する。

1. プレイアウト速度確認
2. 候補打牌比較器
3. 軽い探索
4. MCTS 併用学習

### 12.2 初期探索の目的
初期探索では、まず以下を確認する。

- プレイアウト速度
- 同一局面に対する打牌候補比較の安定性
- ベースラインとの比較可能性

### 12.3 MCTS の位置づけ
MCTS は最終的な重要要素だが、**最初の段階で必須とはしない**。  
まずは学習器とプレイアウト基盤の成立を優先する。

### 12.4 蒸留実験
Stage 2 候補として、**FullObservation teacher → PartialObservation student** の蒸留実験を明記する。  
これにより、完全情報で学習しやすいモデルから、不完全情報で使えるモデルへの知識移送を試みる。

---

## 13. 学習データ方針

### 13.1 基本単位
学習データは **ステップ単位** を基本とする。  
ただし、局・半荘の境界情報を保持する。

### 13.2 1 サンプルに含める情報
少なくとも以下を含めることを想定する。

- Observation
- legal action mask
- chosen action
- reward
- terminal flags
- episode / round / step metadata

### 13.3 保存方針
- ローカル検証だけでなく、将来の分散 self-play を見据える
- データは file-based shard 保存を前提とする
- 保存形式は抽象化し、**推奨実装は Parquet** とする

---

## 14. 実験設定方針

### 14.1 正本
実験設定の人間可読な正本は **YAML** とする。

### 14.2 初期運用
初期運用は **1実験1YAML** を推奨する。  
将来的には base + override や設定分割へ拡張してよい。

### 14.3 設定可能にすべき項目
- 観測形式
- encoder
- model
- reward
- self-play 相手比率
- replay/shard 設定
- 学習ハイパーパラメータ
- 評価設定

---

## 15. 実験記録方針

### 15.1 基本方針
このプロジェクトでは、実験そのものを研究記録として残す。  
そのため、モデル重みだけでなく、**設定・指標・メモ・評価結果** をセットで保存する。

### 15.2 実験単位
保存単位は **run 単位** とする。  
実験ディレクトリは以下のような命名を推奨する。

- `runs/<date>_<name>_<id>/`

ここで:
- `<name>` は実験の論理名
- `<id>` は重複回避・連番・短縮IDなどに用いる

### 15.3 実験ディレクトリ構成
各 run ディレクトリは少なくとも以下を持てるようにする。

- `config.yaml`
- `notes.md`
- `checkpoints/`
- `selfplay/`
- `eval/`

### 15.4 notes
- 各実験ディレクトリに `notes.md` を持つ
- `config.yaml` 側にも簡単な memo フィールドを持ってよい
- 実行時に YAML の memo や主要設定が `notes.md` に転写される仕組みを推奨する

### 15.5 checkpoint 比較
checkpoint ごとの比較は、実験ごとの notes とは別のログで管理してよい。

---

## 16. 評価方針

### 16.1 多面的評価
学習成果は単一指標で判断しない。  
できるだけ多くの指標を保持する。

### 16.2 主指標
主指標は以下を併記する。

- 平均順位
- 平均半荘収支
- 和了率
- 放銃率

### 16.3 補助指標
必要に応じて以下も評価する。

- 平均局収支
- テンパイ率
- 流局率
- 立直率（将来）
- 副露率（将来）
- policy entropy
- value loss
- 学習速度
- プレイアウト速度

### 16.4 比較対象
- ルールベースベースライン
- 過去 checkpoint
- 同一設定の別 seed run
- 必要に応じて Full / Partial 比較

---

## 17. Python / C++ 境界方針

### 17.1 学習
- 学習は **Python + PyTorch** を前提とする

### 17.2 ゲームエンジン
- ゲームエンジンは **C++** を前提とする

### 17.3 境界
- 初期は pybind11 等で Python から C++ engine を呼ぶ方針を推奨する
- 学習ループは Python 側、対局進行は C++ 側を基本とする

### 17.4 将来の高速推論
- 将来的には C++ 側で高速推論を行う
- その第一候補として **ONNX Runtime** を想定する
- そのため、モデル設計は **ONNX export 可能であることを推奨** する

---

## 18. replay / shard / 分散方針

### 18.1 replay buffer
- replay buffer は利用してよい
- ただし、局・半荘境界情報と一緒に扱う

### 18.2 shard
- self-play データは shard file 単位で保存してよい
- 将来の複数 CPU / GPU サーバ運用を見据え、共有ストレージ上で扱える形式を想定する

### 18.3 必須メタデータ
各 shard には少なくとも以下を持たせることを想定する。

- `experiment_id`
- `run_id`
- `worker_id`
- `shard_id`
- `model_version`
- `generation`
- `timestamp`
- `episode_id`
- `round_id`
- `step_id`

---

## 19. Full → Partial の関係

### 19.1 最終目標
最終目標は PartialObservation での実戦的推論である。

### 19.2 Full の利用価値
ただし、FullObservation は以下の価値を持つ。

- 学習基盤の立ち上げ
- 上限比較
- teacher モデルの学習
- 蒸留実験

### 19.3 蒸留
Stage 2 候補として、**FullObservation teacher から PartialObservation student への蒸留** を明記する。  
この実験が不可能にならないよう、初期から Observation / Encoder / Model の分離を前提とする。

## 20. RL 側の変更管理方針

### 20.1 CHANGE_QUEUE の扱い
RL 側の変更も、ゲームエンジン側と同じ `CHANGE_QUEUE.md` で管理する。

### 20.2 Type 拡張
RL 側の変更要求では、必要に応じて以下の Type を使ってよい。

- `RL`
- `Training`
- `Experiment`
- `Eval`

### 20.3 原則
- 学習方針の変更は `RL_RULE.md` に反映されるべきである
- 学習システム実装の変更は `RL_SPEC.md` に反映されるべきである
- 実装と最終的に一致していなければならない正本は `RL_SPEC.md` とする

## 21. 将来拡張

将来的に以下を段階的に追加する。

- PartialObservation を前提とした本格学習
- 蒸留実験
- MCTS 併用学習

- 完全ルール対応

- より高度な reward / curriculum / league 学習

## 22. この文書の解釈原則

- この文書は「学習問題をどう定義するか」を定める
- 実装詳細は `RL_SPEC.md` で定める
- 実験で比較可能性を失わないことを重視する
- 最終目標は完全行動・完全ルールであるが、初期段階では段階的簡略化を許可する
- 研究上の柔軟性を残しつつ、推奨初期値と推奨経路は明記する
